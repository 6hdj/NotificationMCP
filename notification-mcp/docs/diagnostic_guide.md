# 一键故障排查 — 完整技术文档

> 适用版本：notification-mcp v1.0.0+
> 最后更新：2026-04-28

---

## 一、功能概述

一键故障排查是通知读取 MCP 工具的内置诊断模块，当服务中断、通知收不到、权限异常等问题发生时，自动执行全链路排查，精准定位根因，输出可执行的修复方案。

### 核心能力

| 能力 | 说明 |
|------|------|
| 全链路自动诊断 | 6 大诊断链路、20+ 检查项，覆盖从权限到日志的完整链路 |
| 精准根因定位 | 不是笼统的"检查权限"，而是精确到哪个权限、什么状态、如何修复 |
| 一键自动修复 | 自动跳转到对应的系统设置页面，用户只需确认即可 |
| 诊断历史追踪 | 保留最近 10 次诊断记录，可对比趋势 |
| 分类诊断 | 支持按类别单独诊断（权限/服务/系统/连接/资源/日志） |

---

## 二、六大诊断链路

### 链路 1：权限校验（5 项）

| 检查项 | 说明 | 失败后果 | 修复方式 |
|--------|------|---------|---------|
| 通知使用权 | 核心权限，监听系统通知 | 完全无法读取通知 | 跳转通知使用权设置 |
| 通知发送权限 | Android 13+ 运行时权限 | 无法显示状态通知和告警 | 跳转通知设置 |
| 电池优化白名单 | Doze 模式白名单 | 服务被冻结或杀死 | 弹出电池优化对话框 |
| 自启动权限 | MIUI/HyperOS 特有 | 重启后无法自动运行 | 跳转自启动管理 |
| 后台弹出界面 | MIUI/HyperOS 特有 | 后台告警无法显示 | 跳转权限管理 |

### 链路 2：服务状态（4 项）

| 检查项 | 说明 | 失败后果 |
|--------|------|---------|
| 通知监听服务 | NotificationListenerService 是否运行 | 无法接收通知 |
| 前台保活服务 | KeepAliveService 是否运行 | 保活能力降低 |
| 守护进程 | WatchdogService 是否运行 | 崩溃后无法自动重启 |
| 崩溃记录 | 今日崩溃次数 | 服务不稳定 |

### 链路 3：系统拦截（5 项）

| 检查项 | 说明 | 影响 |
|--------|------|------|
| Doze 模式 | 设备是否处于 Doze 状态 | 后台网络和任务被限制 |
| 省电模式 | 系统省电模式是否开启 | 后台活动和网络被限制 |
| MIUI 电池策略 | MIUI 后台限制 | 进程被冻结 |
| 神隐模式 | MIUI 14+ 严格后台限制 | 后台活动被严格限制 |
| 可用内存 | 系统内存压力 | 低内存时被 LMK 杀死 |

### 链路 4：连接状态（3 项）

| 检查项 | 说明 |
|--------|------|
| 网络连接 | WiFi/移动数据是否可用 |
| WebSocket 连接 | 与电脑端 Miclaw 的连接状态 |
| 心跳状态 | 心跳包是否正常收发 |

### 链路 5：资源状态（3 项）

| 检查项 | 说明 |
|--------|------|
| 未转发通知队列 | 积压的未转发通知数量 |
| 数据库大小 | Room 数据库文件大小 |
| 可用存储空间 | 设备剩余存储空间 |

### 链路 6：日志分析（2 项）

| 检查项 | 说明 |
|--------|------|
| 错误日志分析 | 扫描最近 100 条 logcat 中的异常关键词 |
| 系统杀进程事件 | 统计系统杀进程日志中的本应用记录 |

## 三、使用方式

### 方式 1：电脑端 Miclaw 对话（推荐）

直接在电脑端 Miclaw 中说：

```
排查通知读取服务的问题
```

或更具体地：

```
诊断通知读取服务的权限状态
```

Miclaw 会自动调用 `run_diagnostic` 工具，返回完整诊断报告。

**分类诊断示例：**

```
只检查通知读取的连接状态
```

```
排查通知读取的权限问题
```

### 方式 2：一键修复

诊断完成后，如果发现问题：

```
一键修复通知读取的问题
```

或直接说：

```
自动修复
```

Miclaw 会调用 `auto_fix` 工具，自动跳转到所有需要修复的设置页面。

### 方式 3：手机端 App 界面

打开通知读取 App → 点击「🔍 一键故障排查」→ 查看结果 → 点击「🔧 一键修复」

### 方式 4：查看诊断历史

```
查看通知读取的诊断历史
```

## 四、MCP 工具接口

### 4.1 run_diagnostic

执行全链路诊断。

**参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| category | string | 否 | 诊断类别：`all`(默认) / `permission` / `service` / `system_block` / `connection` / `resource` / `log` |

**返回示例：**

```json

  "overallStatus": "FAIL",
  "summary": "共 22 项检查：18 通过 / 2 警告 / 2 异常",
  "report": "═══════════════════════════════════════\n  📋 通知读取服务 — 全链路诊断报告\n...",
  "checkCount": 22,
  "passCount": 18,
  "warnCount": 2,
  "failCount": 2,
  "fixableCount": 3,

    "跳转到通知使用权设置",

```

一键自动修复所有可修复的问题。

**参数：** 无

**返回示例：**

## 五、诊断报告解读

### 状态含义

| 状态 | 含义 | 处理建议 |
|------|------|---------|
| ✅ PASS | 检查通过 | 无需处理 |
| ⚠️ WARN | 警告 | 不影响核心功能，但建议优化 |
| ❌ FAIL | 异常 | 直接导致服务异常，必须修复 |
| ℹ️ INFO | 提示 | 信息性提示，无需处理 |

### 诊断结论解读

| 结论 | 含义 |
|------|------|
| 所有检查通过，服务运行正常 | ✅ 无需任何操作 |
| 发现 N 个警告，服务基本正常但建议优化 | ⚠️ 核心功能正常，可选择性优化 |
| 发现 1 个关键问题: XXX — YYY | ❌ 精确指出问题和修复方法 |
| 发现 N 个关键问题，服务可能无法正常工作 | ❌ 需要立即修复 |

## 六、常见诊断场景

### 场景 1：通知完全收不到

**典型诊断结果：**
- ❌ 通知使用权：未授权
- ❌ 通知监听服务：未运行

**修复步骤：**
1. 说「一键修复」
2. 在弹出的「通知使用权」设置页面中，找到「通知读取」并开启
3. 确认后服务自动重启

### 场景 2：通知能收到但电脑端看不到

**典型诊断结果：**
- ✅ 通知使用权：已授权
- ✅ 通知监听服务：正在运行
- ❌ 网络连接：无网络
- ❌ WebSocket 连接：未连接

**修复步骤：**
1. 检查手机网络连接
2. 确认电脑端 Miclaw 已启动
3. 说「一键修复」重连 WebSocket

### 场景 3：服务运行几小时后自动停止

**典型诊断结果：**
- ❌ 电池优化白名单：未加入
- ⚠️ MIUI 电池策略：可能限制后台
- ⚠️ 系统杀进程事件：发现 5 次

**修复步骤：**
1. 说「一键修复」
2. 在弹出的电池优化页面中选择「无限制」
3. 参考 MIUI 保活优化指南完成所有保活设置

### 场景 4：重启手机后服务不自动启动

**典型诊断结果：**
- ⚠️ 自启动权限：未允许
- ⚠️ 守护进程：未运行

**修复步骤：**
1. 说「一键修复」
2. 在自启动管理中开启「通知读取」的自启动权限
3. 重启手机验证

## 七、技术实现

### 架构

```
DiagnosticTool (MCP 接口层)
    ↓
DiagnosticEngine (诊断引擎)
    ↓
┌─────────────────────────────────────┐
│  链路1: checkPermissions()          │
│  链路2: checkServiceStatus()        │
│  链路3: checkSystemBlocking()       │
│  链路4: checkConnection()           │
│  链路5: checkResources()            │
│  链路6: analyzeLogs()               │
└─────────────────────────────────────┘
    ↓
DiagnosticReport (报告生成)
    ↓
  保存历史 + 返回结果
```

### 文件结构

```
diagnostic/
├── DiagnosticReport.kt    # 数据模型（CheckItem, CheckCategory, DiagnosticReport）
├── DiagnosticEngine.kt    # 诊断引擎（六大链路 + 一键修复）
└── DiagnosticTool.kt      # MCP 工具接口层
```

### 依赖关系

- `DiagnosticEngine` 依赖 `AppDatabase` 查询通知队列和数据库大小
- `DiagnosticEngine` 依赖 `McpWebSocketManager` 查询连接和心跳状态
- `DiagnosticTool` 依赖 `DiagnosticEngine` 执行诊断和修复
- UI 层 `DiagnosticActivity` 依赖 `DiagnosticEngine` 展示结果

### 性能考量

- 全链路诊断耗时 < 3 秒（主要耗时在 logcat 读取）
- 分类诊断耗时 < 1 秒
- 诊断历史存储在 SharedPreferences，不占用数据库资源
- 最多保留 20 条历史记录，自动清理旧数据

## 八、扩展指南

### 添加新的检查项

1. 在 `DiagnosticEngine.kt` 对应的链路方法中添加检查逻辑
2. 构造 `CheckItem` 对象并添加到 `checks` 列表
3. 如果支持自动修复，设置 `fixIntent` 字段

示例：

```kotlin
private fun checkNewFeature() {
    val status = /* 检查逻辑 */
    checks.add(CheckItem(
        name = "新功能检查",
        category = CheckCategory.PERMISSION,  // 或其他类别
        status = if (status) CheckStatus.PASS else CheckStatus.WARN,
        detail = if (status) "正常" else "需要修复",
        fixAction = if (!status) "修复方法描述" else null,
        fixIntent = if (!status) "android.settings.SOME_SETTINGS" else null
    ))
}
```

### 添加新的诊断类别

1. 在 `DiagnosticReport.kt` 的 `CheckCategory` 枚举中添加新类别
2. 在 `DiagnosticEngine.kt` 中添加对应的 `checkXxx()` 方法
3. 在 `runFullDiagnostic()` 和 `runCategoryDiagnostic()` 中调用
4. 在 `mcp_config.json` 的 `run_diagnostic` 工具的 `category` 枚举中添加新值
